## 一、核心问题诊断

通过深度代码分析（2600+ 行核心服务代码），发现当前压缩系统存在**五大致命瓶颈**：

### 1. 单级压缩刚性 ⚠️
- **现状**: Agent A 固定 8-15% 压缩率
- **问题**: 无法适应不同场景（导出 vs 存档 vs 周报）
- **影响**: 过度压缩导致信息丢失，或压缩不足浪费空间

### 2. 质量黑盒 ⚠️
- **现状**: 无质量评估机制
- **问题**: 压缩失败时用户无感知，输出质量无保证
- **影响**: 用户信任度下降，需要手动检查结果

### 3. 重复计算浪费 ⚠️
- **现状**: 零缓存，相同对话重复压缩
- **问题**: 每次导出都重新计算，浪费 LLM 调用
- **影响**: 成本高、延迟高、用户体验差

### 4. 长文本脆弱 ⚠️
- **现状**: 超过 12KB 直接截断
- **问题**: 无分块策略，信息丢失严重
- **影响**: 长对话无法完整压缩

### 5. 导出体验割裂 ⚠️
- **现状**: Compact 模式仅截取前 3 问 + 最后 800 字符
- **问题**: 非真正压缩，中间内容全部丢失
- **影响**: 导出质量差，无法满足实际需求

## 三、技术方案：PACS 架构

### 3.1 整体架构

**PACS (Production Adaptive Compression System)** - 生产级自适应压缩系统

```
┌─────────────────────────────────────────────────────────┐
│  Layer 0: Smart Router (智能路由)                       │
│  • 场景识别: 导出/摘要/周报/存档                         │
│  • 策略选择: 自动选择最优压缩级别                        │
│  • 缓存检查: 命中则直接返回                              │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  Layer 1: Semantic Chunking (语义分块)                  │
│  • 本地算法，<100ms，无 LLM 调用                        │
│  • 压缩率: 60-80%                                       │
│  • 用途: 快速预览、邮件分享                              │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  Layer 2: Hierarchical Summary (分层摘要)               │
│  • Map-Reduce 模式，~500ms，1-3 次 LLM 调用            │
│  • 压缩率: 15-25%                                       │
│  • 用途: 导出 Notion、粘贴到 Claude、对话摘要           │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  Layer 3: Knowledge Graph (知识图谱)                    │
│  • 结构化抽取，~1s，1 次 LLM 调用                       │
│  • 压缩率: 3-8%                                         │
│  • 用途: 周报生成、Network 节点、宏观洞察               │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  Quality Gate: 四维质量评估                             │
│  • 语义相似度 + 信息完整性 + 结构完整性 + 可读性        │
│  • 不达标时自动降级                                      │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  Compression Cache: 三级缓存                            │
│  • L1: Memory (50 条, <10ms)                           │
│  • L2: IndexedDB (10000 条, <50ms)                     │
│  • L3: LocalStorage (100 条, <20ms)                    │
└─────────────────────────────────────────────────────────┘
```

### 3.2 核心创新点

#### 1. 智能路由 (Smart Router)
- **场景感知**: 根据导出目标、对话长度、用户偏好自动选择压缩级别
- **自适应**: 短对话降级到 Tier 1，超长对话强制分块
- **缓存优先**: 命中缓存时直接返回，节省 100% 计算

#### 2. 语义分块 (Semantic Chunker)
- **智能分块**: 基于对话轮次 + 语义相似度聚类
- **重要性评分**: 实体密度、问题标记、决策标记、代码块
- **纯本地**: 无 LLM 调用，<100ms 完成

#### 3. 分层摘要 (Hierarchical Summarizer)
- **Map-Reduce**: 并行压缩每个块，树形归并
- **复用 Agent A**: 保持现有 Prompt 和输出格式
- **质量保证**: 每一层都有质量检查

#### 4. 知识图谱 (Knowledge Graph Extractor)
- **结构化**: 实体 + 关系 + 时间线 + 决策 + 问题
- **可视化**: 支持未来 Network 可视化
- **可查询**: 支持关系查询和推理

#### 5. 质量评估 (Quality Gate)
- **四维评分**: 语义相似度、信息保留率、结构完整性、可读性
- **自动降级**: 不达标时回退到上一级
- **可追溯**: 每次压缩都有质量报告

#### 6. 三级缓存 (Compression Cache)
- **多层缓存**: Memory → IndexedDB → LocalStorage
- **智能失效**: TTL + Prompt 版本 + 源数据哈希
- **高命中率**: 预期 70-85%

## 五、关键技术决策

### 5.1 为什么选择三级架构？

| 级别 | 用途 | 优势 | 劣势 |
|------|------|------|------|
| **Tier 1** | 快速预览 | 极速、无成本 | 压缩率低 |
| **Tier 2** | 日常使用 | 平衡质量与速度 | 需要 LLM |
| **Tier 3** | 深度分析 | 结构化、可查询 | 延迟高 |

**结论**: 三级架构覆盖所有场景，用户可根据需求选择。

### 5.2 为什么需要质量评估？

**问题**: 当前压缩失败时用户无感知，导致信任度下降。

**解决**: 四维评分 + 自动降级，确保输出质量始终可控。

### 5.3 为什么需要缓存？

**数据**:
- 用户平均每天导出 3-5 次相同对话
- 周报生成需要压缩 20-50 个对话
- 重复压缩浪费 60-80% 的 LLM 调用

**收益**: 缓存命中率 70-85%，节省大量成本和时间。
